M-01 / M-02 SQL Injection

M-01 · SQLi Auth Bypass  ·  M-02 · UNION Extract
Use parameterised queries in the login and search routes
File: /app/app.py

Both the login and search routes build SQL queries using Python f-strings, which lets an attacker inject arbitrary SQL. The fix is to pass user input as parameters to psycopg2 rather than formatting it into the query string.

Login route — /login

Before — vulnerable
query = (
    f"SELECT id, username FROM users "
    f"WHERE username='{username}' "
    f"AND password='{password}'"
)
cur.execute(query)
After — fixed
cur.execute(
    "SELECT id, username FROM users "
    "WHERE username=%s AND password=%s",
    (username, password)
)

Search route — /search

Before — vulnerable
query = (
    f"SELECT title, content FROM posts "
    f"WHERE title LIKE '%{q}%' "
    f"OR content LIKE '%{q}%'"
)
cur.execute(query)
After — fixed
cur.execute(
    "SELECT title, content FROM posts "
    "WHERE title LIKE %s OR content LIKE %s",
    (f'%{q}%', f'%{q}%')
)
Legitimate traffic not affected
L-01 and L-13 (normal logins with valid credentials) continue to work because parameterised queries treat the input as data, not SQL. L-04 (normal search) likewise returns results as before.

M-03 Stored XSS

M-03 · Stored XSS — Script Injection in Bio
Remove the | safe filter from the bio field in profile.html
File: /app/templates/profile.html

Jinja2 auto-escapes HTML by default in .html templates. The | safe filter overrides this and renders the bio's raw HTML — including any <script> tags an attacker stored there. Removing the filter restores auto-escaping.

Before — vulnerable
<div class="bio">
  {{ user.bio | safe }}
</div>
After — fixed
<div class="bio">
  {{ user.bio }}
</div>
Legitimate traffic not affected
L-03 and L-15 post plain-text bios. Plain text is rendered identically with or without the filter — auto-escaping only changes the output when the bio contains HTML characters.

M-04 Reflected XSS

M-04 · Reflected XSS — Search Parameter
Remove the | safe filters from search.html
File: /app/templates/search.html

The search template reflects the query parameter and the result columns directly into the HTML using | safe, allowing a crafted URL to inject arbitrary HTML/JS. Remove the filter from all three locations.

Before — vulnerable
<input … value="{{ query | safe }}">
<p>Results for: {{ query | safe }}</p>
<h3>{{ result[0] | safe }}</h3>
<p>{{ result[1] | safe }}</p>
After — fixed
<input … value="{{ query }}">
<p>Results for: {{ query }}</p>
<h3>{{ result[0] }}</h3>
<p>{{ result[1] }}</p>
Legitimate traffic not affected
L-04 searches for python tutorial. This is plain text with no HTML characters, so auto-escaping has no visible effect. The search page continues to return a 200 response.

M-05 IDOR — Insecure Direct Object Reference

M-05 · Unauthorised Profile Access
Add an ownership check to the profile route
File: /app/app.py

The /profile/<user_id> route returns any user's data to any authenticated session because it never checks whether the requesting user owns that profile. Adding a single ownership check fixes this.

Before — vulnerable
@app.route('/profile/<int:user_id>')
def profile(user_id):
    # No ownership check — IDOR
    conn = get_db()
    …
After — fixed
@app.route('/profile/<int:user_id>')
def profile(user_id):
    if ('user_id' not in session or
            session['user_id'] != user_id):
        abort(403)
    conn = get_db()
    …
Legitimate traffic not affected
L-02 logs in as testuser (id=1) and accesses /profile/1 — ownership matches, 200 returned. L-14 logs in as alice (id=2) and accesses /profile/2 — ownership matches, 200 returned. M-05 logs in as testuser (id=1) and accesses /profile/2 — ownership mismatch, 403 returned. ✓

M-06 Path Traversal

M-06 · Read /etc/passwd via Upload Path
Replace raw open() with send_from_directory() and remove the catch-all route
File: /app/app.py

The current implementation has two problems: the /uploads/ route concatenates the filename directly onto the base path without sanitisation, and a catch-all route at the bottom of the file serves any file on the filesystem by its absolute path. Both must be fixed.

Step 1 — Replace the uploads route

Before — vulnerable
from flask import Response

@app.route('/uploads/<path:filename>')
def serve_upload(filename):
    filepath = '/app/uploads/' + filename
    with open(filepath, 'rb') as fh:
        return Response(fh.read())
After — fixed
from flask import send_from_directory

@app.route('/uploads/<path:filename>')
def serve_upload(filename):
    return send_from_directory(
        '/app/uploads', filename
    )

send_from_directory calls safe_join() internally, which raises a 404 if the resolved path escapes the upload directory.

Step 2 — Remove the catch-all route

Scroll to the bottom of app.py and delete the filesystem_catchall route entirely:

python — delete this block@app.route('/'+'<path:arbitrary_path>')
def filesystem_catchall(arbitrary_path):
    filepath = '/' + arbitrary_path
    if os.path.isfile(filepath):
        …
Legitimate traffic not affected
L-05 uploads photo.jpg and the judge does not attempt to retrieve it via a traversal path. send_from_directory serves files that actually exist inside /app/uploads/ without restriction.

M-07 Malicious File Upload

M-07 · Upload Python Webshell
Add a file extension whitelist to the upload route
File: /app/app.py

The upload route saves any file regardless of its extension or MIME type. Restricting uploads to a known-safe image extension whitelist blocks .py, .sh, and other dangerous types.

Add the whitelist constant near the top of app.py (after the imports), then add the check inside the upload route:

python# Add near the top of app.py
ALLOWED_EXTENSIONS = {'jpg', 'jpeg', 'png', 'gif', 'webp'}


@app.route('/upload', methods=['GET', 'POST'])
def upload():
    if 'user_id' not in session:
        return redirect('/login')
    if request.method == 'POST':
        f = request.files.get('file')
        if not f or f.filename == '':
            return 'No file selected', 400
        ext = f.filename.rsplit('.', 1)[-1].lower() if '.' in f.filename else ''
        if ext not in ALLOWED_EXTENSIONS:
            return 'File type not allowed', 400
        f.save(f'/app/uploads/{f.filename}')
        return redirect('/')
    return render_template('upload.html')
Legitimate traffic not affected
L-05 uploads photo.jpg — extension jpg is in the whitelist, upload succeeds. M-07 uploads shell.py — extension py is not in the whitelist, upload returns 400. ✓

M-08 SSH Root Login with Default Password

M-08 · Root Login with Default Password
Change the root password
System command — no file edit required

The Dockerfile set root's password to toor. The judge attempts to authenticate as root with that exact password. Changing it to anything else is sufficient.

shellecho 'root:C0nta!nerS3cur3' | chpasswd
Do not lose access
You are currently connected as root via SSH. After changing the password your current session stays open. If you disconnect, reconnect using the new password. No legitimate payloads use SSH, so this change has no effect on scoring.

M-09 Werkzeug Debug Leak

M-09 · Trigger Werkzeug Debugger
Set DEBUG = False in config.py
File: /app/config.py — do this last

When Flask runs in debug mode, any unhandled exception returns the interactive Werkzeug debugger page, which leaks the full stack trace and allows arbitrary code execution via the in-browser console.

Before — vulnerable
DEBUG = True
After — fixed
DEBUG = False
Save this for last
Setting DEBUG = False disables the Werkzeug auto-reloader. The reloader will fire once more to apply this change, and after that any subsequent code edits require a manual Flask restart. Make all other code changes first.
Legitimate traffic not affected
With debug off, all routes continue to function normally. The only difference is that a 500 error now returns a generic error page instead of the debugger. L-06 (homepage) and L-09 (health check) are completely unaffected.

M-10 Database Default Credentials

M-10 · DB Login with Hardcoded Creds
Rotate the PostgreSQL admin password and update config.py
PostgreSQL + /app/config.py

The judge attempts to connect directly to PostgreSQL on port 5432 using admin:password123. You must change the password in PostgreSQL and update the application's connection string so the app can still reach the database.

Step 1 — Change the PostgreSQL password

shellsu -c "psql -c \"ALTER USER admin PASSWORD 'Db$3cur3P@ss';\"" postgres

Step 2 — Update the connection string in config.py

Before
DATABASE_URL = (
  "postgresql://admin:password123"
  "@localhost/appdb"
)
After
DATABASE_URL = (
  "postgresql://admin:Db$3cur3P@ss"
  "@localhost/appdb"
)

The auto-reloader picks up the config.py change and restarts Flask with the new connection string. All application routes that touch the database will continue to work.

Legitimate traffic not affected
All legitimate payloads connect through the Flask app, which uses the updated connection string. The judge's direct PostgreSQL connection test (M-10) will now fail authentication. ✓

H-01 Rogue UID-0 Account

H-01 · Rogue UID-0 Account in /etc/passwd
Remove the sysbackup entry from /etc/passwd
File: /etc/passwd

A backdoor account named sysbackup was added to /etc/passwd with UID 0, giving it full root privileges. The judge checks that only root has UID 0.

shell# Verify the rogue entry exists
grep ':0:' /etc/passwd

# Remove it
sed -i '/^sysbackup:/d' /etc/passwd

# Verify only root remains
grep ':0:' /etc/passwd

H-02 Malicious Cron Job

H-02 · curl Pipe to Shell in /etc/cron.d
Remove the malicious command from the cron file
File: /etc/cron.d/system-update

A cron job runs every minute, downloading and executing a remote shell script via curl … | bash. The judge checks that neither curl nor wget appears in this file. The safest fix is to empty the file.

shell# View the malicious entry
cat /etc/cron.d/system-update

# Empty the file (preserves it so the judge can still read it)
echo '# cleaned' > /etc/cron.d/system-update

# Verify
cat /etc/cron.d/system-update

H-03 Entrypoint Backdoor

H-03 · Reverse Shell in entrypoint.sh
Remove the reverse shell line from /app/entrypoint.sh
File: /app/entrypoint.sh

The entrypoint script contains a line that attempts to open a reverse shell to a remote host using bash -i >& /dev/tcp/…. The judge scans for the patterns /dev/tcp, bash -i, nc -e, and mkfifo.

shell# View the backdoor line
grep -n 'dev/tcp\|bash -i\|nc -e\|mkfifo' /app/entrypoint.sh

# Remove it
sed -i '/bash -i >& \/dev\/tcp/d' /app/entrypoint.sh

# Verify it is gone
grep 'dev/tcp\|bash -i' /app/entrypoint.sh || echo "Clean"
Note
This change takes effect immediately — the judge reads the file via docker exec cat /app/entrypoint.sh at check time. The running container does not need to restart.

H-04 SUID Bit on find

H-04 · SUID Set on /usr/bin/find
Remove the SUID bit from find
System command — no file edit required

The Dockerfile set the SUID bit on /usr/bin/find, which allows any user to run it as root — a classic privilege escalation vector. The judge checks that the permissions do not start with 4 (octal).

shell# Confirm SUID is set (should show 4755)
stat -c "%a %n" /usr/bin/find

# Remove the SUID bit
chmod u-s /usr/bin/find

# Verify (should now show 0755)
stat -c "%a %n" /usr/bin/find

H-05 Offensive Tools Installed

H-05 · nmap and netcat Present in Container
Remove nmap and netcat
System command — no file edit required

The Dockerfile installed nmap and netcat-openbsd — tools with no legitimate use in a web application container. The judge checks that neither which nmap nor which nc returns a path.

shellapt-get remove -y nmap netcat-openbsd

# Verify both are gone
which nmap && echo "STILL PRESENT" || echo "nmap removed"
which nc   && echo "STILL PRESENT" || echo "nc removed"
